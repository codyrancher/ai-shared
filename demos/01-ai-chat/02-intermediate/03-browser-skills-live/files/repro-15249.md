Here's a summary of the reproduction and findings:

## Reproduction Steps

1. **Apply a CRD with inline nested spec definitions** (object fields with `properties` defined directly, not via `$ref`). I created a `TestCertificate` CRD at `example.io` with nested fields like `issuerRef`, `privateKey`, and `additionalOutputFormats`, each with descriptions.
2. **Create an instance** of that CRD (`test-cert-1`).
3. **Navigate to the CRD list** in Explorer: `More Resources > example.io > TestCertificates`.
4. **Click the Explain header button** (book icon in the top toolbar).
5. **Observe**: `spec` shows type "object" with no expand chevron and no nested fields. Same for `status`. The nested field descriptions (`commonName`, `dnsNames`, `issuerRef.name`, `privateKey.algorithm`, etc.) are completely missing.
6. **Contrast**: For built-in resources like Deployments, `spec` shows "DeploymentSpec" with an expand chevron that reveals all nested field descriptions.

## Root Cause

In [open-api-utils.ts:74-104](#browser-skills-live), `expandOpenAPIDefinition()` only resolves nested fields via `$ref` references (`prop.$ref || prop.items?.$ref`). CRDs define their schemas inline in the OpenAPI v2 spec (no `$ref` entries), so their nested properties are never expanded. As the commenter @nwmac noted: "the explain UI does not show object types with properties, where these are declared inline, vs being declared as a schema."
